🎬 AI-DLC 가 잘 맞는 상황 vs 안 맞는 상황

AI-DLC 는 모든 상황에 동일하게 적합한 도구는 아닙니다. 프로젝트의 규모·신/구·변경 성격에 따라 효과가 극단적으로 갈립니다.

이 페이지는 6 가지 대표 시나리오 별로 AI-DLC 가 어떻게 동작하는지, 어떤 설정을 권장하는지, 그리고 적용 시 주의해야 할 한계를 정리합니다.

📊 한눈에 보는 적합도 매트릭스

시나리오별 권장 phase·depth·적합도 요약
# 시나리오 규모 기준 권장 phase 권장 depth 적합도
1 🌱 신규 시스템 구축 (Greenfield) 전 규모 Inception → Construction → Operations standard / comprehensive ✓ 최적
2 🔧 기존 시스템 기능 추가 (소-중규모) ~ 100k LOC Reverse Engineering → Inception → Construction standard ✓ 적합
3 🐛 핫픽스 / 단순 버그 수정 1~수십 줄 변경 (skip 가능) — 직접 수정 minimal ⚠ overkill 가능
4 📦 마이그레이션 / 리아키텍처 중-대규모 Reverse Engineering → Inception (재설계) comprehensive ✓ 매우 적합
5 🔍 기존 코드 분석·문서화 only 중-대규모 Reverse Engineering 만 standard / comprehensive ✓ 적합
6 🏢 초대형 레거시 1,000만+ LOC scope 분할 후 부분 적용 minimal (sub-scope 단위) ✗ as-is 부적합

시나리오별 상세

각 시나리오의 AI-DLC 동작·권장 설정·주의사항
🌱
SCENARIO 1

신규 시스템 구축 (Greenfield)

✓ 최적
상황: 백지 상태에서 새 서비스·MVP·내부 도구를 처음부터 만든다. 기존 코드 제약이 없어 AI-DLC 의 14 step 을 그대로 적용할 수 있다.
AI-DLC 동작
  • Inception 부터 시작 — vision.md · tech-env.md 작성
  • 요구사항 → 사용자 스토리 → 아키텍처 → 컴포넌트 → 유닛 분해까지 자동
  • Brownfield reverse-engineering step 전체 skip
권장 설정
  • depth: standard (MVP) / comprehensive (장기 운영용)
  • Extensions: aws-best-practices, security-baseline 초기부터 opt-in
주의: 너무 작은 prototype 은 minimal depth 로 시작. 모든 산출물을 풀세트로 만들면 학습 부담이 커진다.
🔧
SCENARIO 2

기존 시스템에 기능 추가 (소-중규모)

✓ 적합
상황: 수만~수십만 줄 규모의 기존 코드베이스에 새 endpoint·기능 모듈을 추가한다. 브라우필드의 핵심 use-case.
AI-DLC 동작
  • workspace-detection → 기존 워크스페이스 감지 → reverse-engineering 9 artifact 자동 생성
  • 그 결과를 입력으로 Inception (이번에 추가하는 기능만 한정)
  • Construction 에서 기존 파일 수정 + 신규 파일 생성 혼합
권장 설정
  • depth: standard
  • RE 결과의 code-quality-assessment.md 먼저 검토
  • 변경 범위에 맞춰 scope-limiter Extension 활성화
주의: RE 가 만든 architecture.md 가 실제와 다를 수 있다. Construction 진입 전에 사람이 한 번 review 권장.
🐛
SCENARIO 3

핫픽스 · 단순 버그 수정

⚠ overkill 가능
상황: 1~수십 줄 패치, null 체크 추가, typo, log message 수정 같은 미세 변경.
AI-DLC 동작
  • 워크플로우 전체를 돌리면 requirements / design 산출물이 과도하게 생성됨
  • adaptive workflow 가 일부 step 을 skip 해도 audit / approval 오버헤드는 잔존
권장 설정
  • 워크플로우 진입 없이 직접 수정 + 기존 테스트 실행
  • 변경이 설계에 영향을 준다면 그때 설계 update 만 AI-DLC 패턴으로 진행
  • back-propagation standing rule 로 누락 방지
주의: "버그 같았는데 사실은 잘못된 요구사항" 이라면 즉시 Inception 으로 escalate.
📦
SCENARIO 4

마이그레이션 · 리아키텍처

✓ 매우 적합
상황: 모놀리식 → 마이크로서비스, on-prem → AWS, JS → TS 전환 등 대규모 구조 변경.
AI-DLC 동작
  • RE 로 현 상태 baseline 9 artifact 확보 → 이후 변경의 reference
  • Inception 을 재설계 관점으로 다시 진행 (target architecture)
  • Construction 은 unit 단위 점진 마이그레이션
권장 설정
  • depth: comprehensive (수많은 의사결정 추적 필요)
  • audit.md 는 운영진 보고 자료로 그대로 활용 가능
  • 마이그레이션 phase 별 vision.md 분리 권장
주의: baseline 과 target 두 architecture.md 를 분리 보관. 변경 추적이 단일 파일로 흐려지면 가치가 반감된다.
🔍
SCENARIO 5

기존 코드 분석 · 문서화 (분석 only)

✓ 적합
상황: 인수인계받은 코드, 오래된 시스템의 현황 파악·온보딩 문서 작성. 코드 변경은 하지 않는다.
AI-DLC 동작
  • Reverse Engineering phase 만 수행 → 9 artifact 산출
  • Inception / Construction 으로 진행하지 않고 종료
권장 설정
  • depth: standard (개요) → 필요한 영역만 comprehensive
  • 산출물 그대로 온보딩 가이드 · ADR 인풋으로 활용
주의: RE 산출물은 코드 변경 시점에 stale 이 된다. 주기적 re-run 필요 (workspace-detection 의 stale-artifact 감지 활용).
🏢
SCENARIO 6

초대형 레거시 (1,000만+ LOC)

✗ as-is 부적합
상황: 은행 코어, 통신사 OSS/BSS 같은 수십만 파일·수천 만 줄 규모. AI-DLC 워크플로우의 기본 가정을 벗어난다.
AI-DLC 동작 (그대로 적용 시)
  • RE 의 single architecture.md / code-structure.md 생성이 사실상 불가능 (전 파일 나열·단일 mermaid diagram 요구)
  • session-continuity 가 시작 시 모든 artifact 를 다시 로드 → context window 초과
  • depth-levels 은 디테일만 줄임 — scope 자체는 줄지 않음
  • error-handling.md 에 "codebase too large" 케이스 없음 → 명확한 fail 가이드 부재
권장 우회법
  • 도메인/모듈별 sub-workspace 분할 후 각각 별도 AI-DLC 적용
  • 최상위 architecture.md 는 사람이 high-level 로 미리 작성·고정
  • AI 는 한 sub-scope 의 unit-detail 까지만 담당
  • Extension 으로 scope-limiter 강제 — RE 가 전 코드 스캔하지 못하게 차단
주의: 1,000만 LOC ≈ 2억+ 토큰. 어떤 LLM context window 로도 통째 분석 불가. AI-DLC 는 인간이 scope 를 분할해 준 후에 비로소 정상 동작한다. 자세한 한계·대응은 아래 deep-dive 참고.

🏢 Deep Dive — 1,000만 LOC 레거시에서 무엇이 무너지는가

SCALE LIMIT AI-DLC workflow rule 별 구체적 blocker 와 대응법

AI-DLC 의 룰 (aidlc-rules/aws-aidlc-rule-details/) 을 정밀히 읽어 보면, 모든 산출물이 "전 코드베이스 단일 view" 를 전제로 설계되어 있습니다. 이 전제가 깨지는 시점이 약 수십만 LOC 부근이며, 1,000만 LOC 에서는 5 가지 구조적 blocker 가 동시에 발생합니다.

🚧 구조적 Blocker
aidlc-rules/inception/reverse-engineering.md

1. code-structure.md — 모든 소스 파일 나열 요구

RE rule 은 "list all source files with their purpose" 를 요구합니다. 5만+ 파일에서는 산출물 자체가 수십 MB 가 되고 LLM 이 한 번에 다룰 수 없습니다.

aidlc-rules/inception/reverse-engineering.md

2. architecture.md — 단일 Mermaid diagram 요구

"all major packages and their relationships" 를 하나의 mermaid 그래프로 그리도록 명시. 1만+ 모듈에서는 가독성 0, 렌더링 자체도 실패.

aidlc-rules/inception/workspace-detection.md

3. workspace scan — 샘플링·상한 없음

워크스페이스 감지 시 전 파일 스캔이 기본. cap·sampling 룰이 없어 초대형 repo 에서는 detection 자체가 다 끝나기 전에 토큰이 소진됩니다.

aidlc-rules/common/session-continuity.md

4. session resume — 모든 artifact 재로딩

세션 재개 시 aidlc-docs/ 의 모든 산출물을 다시 읽어들임. artifact 가 거대해지면 매 resume 마다 context 의 대부분이 잠식됨.

aidlc-rules/common/depth-levels.md

5. depth 조정 ≠ scope 조정

minimal / standard / comprehensive 는 각 항목의 설명 디테일만 조정합니다. "어느 모듈만 다룰지" 의 scope 자체를 줄여 주는 룰은 존재하지 않습니다.

aidlc-rules/common/error-handling.md

6. "too large" 케이스 없음

error-handling rule 에 codebase 가 너무 커서 처리 불가한 상황이 정의되어 있지 않아, AI 가 시도를 계속하다 token-limit 으로 abort 됩니다 — graceful degradation 없음.

🛠️ 실전 우회 전략
STRATEGY 1

도메인 단위 sub-workspace 분할

모놀리식 repo 안에서도 도메인별 디렉토리를 각각 별도 워크스페이스로 취급. 각 sub-workspace 마다 독립된 aidlc-docs/ 를 둔다.

STRATEGY 2

최상위 architecture.md 를 사람이 사전 고정

RE 가 만들지 않고, 사람이 high-level map 을 먼저 작성해 둔다. AI 는 그 안의 한 노드(sub-domain) 만 detail 화하는 역할로 한정.

STRATEGY 3

scope-limiter / file-glob extension 활성화

Extension 메커니즘으로 RE 가 스캔할 파일 범위를 명시적으로 제한. 예: include: ["domain/billing/**"] · exclude: ["vendor/**","legacy/**"].

STRATEGY 4

session 단위 artifact 분리 보관

aidlc-docs/<sub-scope>/ 처럼 폴더 분리. session-continuity 가 resume 시 해당 sub-scope 의 artifact 만 로드하도록 운영.

💡 요약 — AI-DLC 가 1,000만 LOC 에서 "안 된다" 는 것은 모델 한계가 아니라 워크플로우 룰이 단일 view 전제로 쓰여 있기 때문입니다. 운영자가 scope 를 잘라 주면 AI-DLC 의 가치(추적성·일관성·Human-in-the-loop)는 그대로 유지됩니다.
