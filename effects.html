AI-DLC 란? AI Development Life Cycle

AI-DLC 는 AWS Labs 가 만든 AI 코딩 에이전트(Claude Code / Cursor / Q Developer 등)를 위한 표준 개발 워크플로우입니다. "vibe coding" 처럼 즉흥적으로 코드를 생성하는 대신, 요구사항 수집 → 설계 → 코드 생성 → 빌드/테스트 의 전 과정을 구조화된 단계로 진행하며, 모든 단계마다 사용자가 명시적으로 검토·승인할 수 있는 Human-in-the-loop gate 를 둡니다.

PRINCIPLE 1
📋 File-based questions
AI 는 채팅으로 질문하지 않고, multiple-choice 질문 파일에 [Answer]: 태그를 남겨 사용자가 파일에 답변하도록 합니다. 모호함이 체계적으로 제거됩니다.
PRINCIPLE 2
📝 Audit trail by default
모든 결정·승인·응답이 audit.md 에 ISO 8601 timestamp 와 함께 append-only 로 기록되어, 누가 언제 무엇을 승인했는지 항상 추적됩니다.
PRINCIPLE 3
🔀 Adaptive workflow
프로젝트의 복잡도·greenfield/brownfield 여부에 따라 14개 step 중 일부가 자동 skip 되거나 depth (minimal/standard/comprehensive) 가 조정됩니다.

INCEPTION

→
PHASE 2

AI 의 도움으로 얻는 효과

AI Capability AI 코딩 에이전트가 실제로 처리하는 작업 영역과 그 결과로 얻는 가치
📋
REQUIREMENTS

요구사항 분석

사용자 요청을 product owner 관점으로 받아 6 영역 완성도 분석(Functional / NFR / User Scenarios / Business Context / Technical Context / Quality Attributes) 을 자동 수행합니다.
  • 요청 명확성·복잡도 자동 판별 (clear / vague / incomplete)
  • 모호한 응답·답변 충돌 자동 탐지 → clarification 라운드
  • 구조화된 multiple-choice 질문 자동 생성
🏗️
DESIGN

아키텍처·컴포넌트 설계

high-level 컴포넌트와 service layer 를 자동 식별하고, 의존성 그래프와 unit-of-work 까지 분해합니다.
  • components / methods / services / dependency 4개 산출물 자동 생성
  • 병렬 개발 가능한 unit-of-work 로 자동 분해
  • DDD aggregate / circular dependency 자동 검출
💻
CODE GENERATION

코드 작성

설계 산출물을 입력으로 Unit 단위 병렬 코드 생성. brownfield 는 기존 파일 수정, greenfield 는 신규 생성.
  • 2-phase (plan → execute) 로 코드 생성 전 사용자 검토
  • 일관된 패턴 적용 (repository / state machine / strategy 등)
  • inline documentation 자동 작성
🧪
TESTING

테스트 자동 생성

unit / integration / performance / contract / security / e2e 테스트 instruction 을 자동 생성하여 빌드·검증 단계를 표준화합니다.
  • NFR 에 따라 조건부 테스트 활성화 (perf / security 등)
  • build-and-test-summary.md 로 전체 결과 통합 리포트
  • 실패 시 자동 디버깅·수정 시도 (real golden sample 192/192 PASS)
📚
DOCUMENTATION

문서화

모든 단계의 산출물이 markdown 으로 자동 누적됩니다. 별도 문서화 작업이 필요 없습니다.
  • aidlc-docs/ 디렉토리에 단계별 산출물 자동 정리
  • 설계 도면(컴포넌트/의존성)을 Mermaid 다이어그램으로 자동 작성
  • API 문서·domain model·business rules 모두 markdown
🔍
BROWNFIELD ANALYSIS

기존 코드베이스 분석

brownfield 프로젝트에 대해 reverse engineering 으로 9개 분석 문서를 자동 생성합니다.
  • business-overview / architecture / API / code-structure 문서화
  • component-inventory / technology-stack / dependencies 식별
  • code-quality-assessment 로 tech debt·테스트 커버리지 평가

AI-DLC 프레임워크 자체가 주는 효과

Framework Value AI 모델 자체와 별개로, AI-DLC 의 워크플로우 구조가 보장하는 효과
🔍
TRACEABILITY

추적성

모든 사용자 결정·승인·AI 응답이 audit.md 에 ISO 8601 timestamp 와 함께 append-only 로 기록됩니다.
  • 누가 언제 무엇을 승인했는지 항상 확인 가능
  • 감사·컴플라이언스 대응에 그대로 사용 가능
  • 잘못된 결정의 원인 분석 (post-mortem) 가능
🎯
HUMAN-IN-THE-LOOP

사용자 결정 권한 보장

AI 가 임의로 진행하지 않도록, 명시적 승인 gate 가 단계 사이마다 배치됩니다.
  • 중요 단계마다 사용자가 검토 후 명시적으로 "Approve" 입력
  • [Answer]: 태그로 사용자가 직접 선택지 입력
  • follow-up clarification 라운드로 모순 해소
🔀
ADAPTIVE

복잡도 적응성

프로젝트 특성에 따라 단계가 자동 skip 되거나 depth 가 조정되어 오버엔지니어링을 방지합니다.
  • 단순 API → User Stories / Unit Generation skip
  • NFR 명세 명확 → NFR Requirements / Design skip
  • 요청 명확성에 따라 minimal / standard / comprehensive 자동 선택
⚠️
CONTRADICTION DETECTION

모순 자동 탐지

사용자 답변 간의 충돌·모호함을 자동 분석하고, *-clarification-questions.md 파일로 후속 질문을 발행합니다.
  • scope/risk/timeline mismatch 자동 감지
  • 해결 전까지 다음 단계 진행 차단 (blocking gate)
  • 일관성 검증 후에만 산출물 생성
🎨
CONSISTENCY

일관된 워크플로우 패턴

모든 step 이 plan → 질문 → 답변 → 검증 → 산출물 생성 의 동일 패턴을 따라 예측 가능한 결과를 보장합니다.
  • 매번 같은 step 순서·산출물 구조
  • 팀원이 산출물 위치를 외울 필요 없음 (표준 경로)
  • 새 unit / phase 추가 시 학습 비용 최소화
🛡️
COMPLIANCE GATES

컴플라이언스 강제

Extensions opt-in 으로 보안·품질·테스트 룰을 blocking constraint 로 적용합니다.
  • opt-in 한 룰은 단계 진행을 막는 강제 조건이 됨
  • aws-best-practices, security-baseline 등 사전 정의 룰
  • 커버리지 임계값·SAST 결과가 자동 검증됨
🤝
COLLABORATION

사람·AI 공동 작업

모든 산출물이 markdown 파일 이라 사람·AI·다른 도구가 동일하게 읽고 쓸 수 있습니다.
  • PR 리뷰·코드 리뷰처럼 산출물도 git diff 가능
  • 중단·이어쓰기 가능 (session-continuity)
  • 여러 AI agent 가 같은 워크스페이스에서 협업
🔌
PORTABILITY

AI 에이전트 독립성

Claude Code / Cursor / Q Developer / Kiro / Amazon Q 등 어느 AI agent 에서 실행해도 같은 워크플로우 룰이 적용됩니다.
  • 도구 변경 시에도 산출물·진행 상태 그대로 이어짐
  • .claude/, .kiro/, .amazonq/ 등 각 도구의 룰 경로 자동 인식
  • vendor lock-in 방지
🔁
REPRODUCIBILITY

재현성

같은 입력 (vision.md / tech-env.md + 답변) → 같은 산출물 구조를 보장합니다.
  • aidlc-evaluator 의 golden sample 기반 검증
  • workflow 가 결정론적 (state machine 기반)
  • 실험·비교·회귀 테스트 가능

오버엔지니어링 AI 가 알아서 추가 추상화·feature flag 생성 요구사항·NFR 명세에 기반해 필요한 만큼만 생성

🧭 왜 의존 산출물 자동 재생성 기능이 없는가?

DESIGN CHOICE "변경 시 의존 md 들을 자동 update" 가 아닌, 사용자가 명시적으로 트리거하는 구조를 의도적으로 선택

AI-DLC 에는 "md 파일 의존성 그래프 → 변경 감지 → 자동 cascade 재생성" 같은 기능이 의도적으로 없습니다. 변경이 발생하면 사용자가 직접 upstream 영향 확인 → 설계 update → 영향 받은 코드만 regenerate 의 3-step 패턴을 트리거합니다. 이는 AI-DLC 의 핵심 tenet (Human in the loop / No duplication / Reproducible) 과 직접 연결된 설계 결정입니다.

REASON 1

🎯 Human-in-the-loop tenet

"Critical decisions require explicit user confirmation" — AI 가 임의로 여러 산출물을 동시 수정하면 사용자가 변경 범위를 인지하지 못한 채 silent drift 가 발생할 수 있습니다.

"The agent proposes, the human approves." — AI-DLC Tenets
REASON 2

🎲 LLM 비결정성 위험

같은 입력이라도 LLM 은 재생성 시 다른 표현·구조를 만들 수 있습니다. 자동 cascade 는 사용자가 이미 승인한 wording 을 임의로 덮어쓸 위험이 있어, audit trail 의 의미를 훼손합니다.

"Rules should be clear enough that different models produce similar outcomes" — 변동성 자체를 최소화하는 방향.
REASON 3

📦 No duplication tenet

"The source of truth lives in one place" — 의존성 그래프를 별도 메타데이터로 관리하면 그 자체가 또 하나의 동기화 대상이 됩니다. AI-DLC 는 markdown 본문이 곧 단일 출처가 되도록 설계합니다.

"We generate them from the source rather than maintaining separate copies."
REASON 4

💰 비용·범위 통제

자동 cascade 는 불필요한 토큰 사용·과도한 재생성으로 이어지기 쉽습니다. 사용자가 영향 범위를 먼저 확인하고 필요한 부분만 regenerate 하면 비용·시간을 통제할 수 있습니다.

"re-run code generation for the affected files only" — 영향 범위를 사용자가 명시.
📌 대신 권장하는 3-step 변경 흐름 docs/WORKING-WITH-AIDLC.md
STEP 1
영향 분석 (no-change)
변경을 가하지 않고 영향 범위를 먼저 파악합니다.
Do not update any documents. Review the design and help me understand where this needs to be addressed.
→
STEP 2
설계 update + upstream 확인
해당 설계 문서 수정 후 상위 산출물 영향도 함께 점검합니다.
Please update [design doc]. Then check whether any upstream documents — requirements, user stories — also need to be updated.
→
STEP 3
영향 코드만 재생성
전체가 아닌 영향 받은 unit/파일에 한해 코드 재생성을 지시합니다.
The design for [unit X] has been updated. Please re-run code generation for the affected files only.
관점 자동 cascade 재생성 (가상) AI-DLC 의 명시적 back-propagation
변경 인지 어느 파일이 함께 바뀌었는지 사후 diff 로 확인 사용자가 단계마다 변경 대상을 직접 승인
결과 안정성 LLM 재생성으로 승인된 wording 이 silent 변경 위험 변경하지 않은 산출물은 그대로 보존 (단일 출처 유지)
감사·추적 자동 cascade 의 일부는 audit 누락 가능 모든 변경 요청이 audit.md 에 append-only 기록
비용·시간 의존 파일 전체 토큰 소비 → 비용 폭증 가능 "affected files only" 로 필요한 부분만 재생성
실수 회복 잘못된 변경이 cascade 로 동시 전파됨 각 step 에서 detect → revert 가능 (게이트 단위 격리)

💡 자동에 가장 근접한 옵션 — Phase 시작 시 standing rule 로 박아둘 수 있습니다 (시스템 기능이 아니라 prompt convention):

STANDING RULE

"Every time you update a document, check whether the change impacts the requirements document and user stories, and prompt me if it does." — 자동 수정은 아니지만 영향 가능성을 매번 알림.
