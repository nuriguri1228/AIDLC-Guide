Reverse Engineering 이란? INCEPTION · BROWNFIELD ONLY

기존 코드베이스를 입력으로 받아, AI-DLC 가 사용할 수 있는 9 개의 분석 산출물 (markdown) 을 생성하는 phase 입니다. 이 산출물들은 이후 Inception · Construction 단계가 참조하는 baseline 역할을 합니다.

Greenfield 에서는 전체 skip 되며, Brownfield 에서는 workspace-detection 이 기존 워크스페이스를 감지한 직후에 트리거됩니다. 모든 산출물은 aidlc-docs/reverse-engineering/ 아래 누적됩니다.

📋 RE 실행 단계 (13 step)

reverse-engineering.md 의 실제 실행 순서
  1. Workspace 감지 workspace-detection.md 룰로 aidlc-docs/ 존재·stale 여부 확인.
  2. Manifest · package descriptor 파싱 언어·프레임워크·빌드툴 식별 (package.json, pom.xml, requirements.txt 등).
  3. Source tree 스캔 디렉토리 구조 · entry point · 주요 모듈 후보 추출.
  4. technology-stack.md 생성 언어·런타임·라이브러리·DB·인프라 분류.
  5. dependencies.md 생성 외부 의존성 + 모듈 간 의존성 + 버전 risk 식별.
  6. code-structure.md 생성 디렉토리 트리 + 각 파일·모듈의 purpose 기록.
  7. component-inventory.md 생성 서비스·클래스·핸들러 단위 component 목록과 책임.
  8. api-inventory.md 생성 HTTP · gRPC · CLI · message handler 등 모든 외부 인터페이스 카탈로그.
  9. architecture.md 생성 전체 모듈 관계를 단일 mermaid diagram + narrative 로 정리.
  10. domain-model.md 생성 엔터티 · aggregate · invariants 식별 (DDD lens).
  11. business-overview.md 생성 비즈니스 목적 · 핵심 use-case · 주요 stakeholder 추정.
  12. code-quality-assessment.md 생성 tech debt · 테스트 커버리지 · 보안 risk · 리팩터 후보 종합 평가.
  13. Audit + HITL gate 9 artifact 완료 후 audit.md 기록 → 사용자 review → Approve 시 다음 step 으로.

파일명 · 목적 · 핵심 contents · downstream 영향
# 파일 목적 핵심 contents 주요 downstream
1 technology-stack.md 기술 스택 카탈로그 언어·런타임·DB·infra · build tool architecture / quality assessment
2 dependencies.md 외부·내부 의존성 트리 패키지 버전 · risk · 모듈 간 dependency architecture / quality assessment
3 code-structure.md 디렉토리·파일 purpose 맵 모든 source file 의 역할 요약 component-inventory / quality
4 component-inventory.md component 단위 책임 정리 service·class·handler 목록 + responsibility architecture / quality / target design
5 api-inventory.md 외부 인터페이스 카탈로그 HTTP/gRPC/CLI/messaging endpoint 목록 domain-model / Inception requirements
6 architecture.md 전체 시스템 구조 정리 단일 mermaid + narrative + key flow domain / target architecture
7 domain-model.md 도메인 엔터티·aggregate entity · invariants · bounded context business-overview / requirements
8 business-overview.md 비즈니스 목적 추정 use-case · stakeholder · 가치 제안 requirements · user-stories
9 code-quality-assessment.md 품질·리스크 종합 평가 tech debt · 커버리지 · 보안 risk · 리팩터 후보 target architecture · 리아키텍처 decision

💡 STANDING RULE — RE 결과를 stale 상태로 두지 않으려면

RE artifact 는 코드가 바뀌는 순간 stale 됩니다. workspace-detection.md 의 stale 감지 룰을 활용해 "code 변경이 감지되면 영향 받은 artifact 를 식별하고 re-run 후보를 알림" 을 standing rule 로 박아 두면, 매번 수동 점검할 필요가 줄어듭니다.

⚠️ 초대형 레거시 (1,000만+ LOC) 에서의 한계

9 artifact 중 code-structure.md 와 architecture.md 는 "전 파일 나열" · "단일 mermaid" 를 전제로 합니다. 이 전제가 깨지는 규모에서는 RE 가 그대로 동작하지 않습니다 — 시나리오 6 의 우회 전략 을 참고하세요.
